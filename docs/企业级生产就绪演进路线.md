# OpenSandbox Plus 企业级生产就绪演进路线

目标：把当前 MVP 演进为企业级生产环境 ready 的云沙箱控制面，具备安全可信、稳定可靠、可观测、可扩展、可运维、可治理的交付能力。

## 演进原则

| 原则 | 说明 |
| --- | --- |
| 先稳后拆 | 当前保持单服务交付，先补齐生产能力；当容量、隔离或发布节奏真正需要时再拆 API/Worker/Scheduler |
| 安全默认开启 | 认证、授权、审计、密钥保护、网络边界和最小权限作为默认设计，不作为后置增强 |
| 兼容优先 | Agent 面向 `/v1/sandboxes` 的 OpenSandbox 兼容 API 保持稳定，避免破坏已有 Agent 集成 |
| 自动化验收 | 每个阶段都必须沉淀可重复运行的 CI、部署检查和回归脚本 |
| 运维可解释 | 平台状态、审计、告警和故障处置流程要能支持值班人员快速定位问题 |

## 阶段 1：生产基线

目标：让当前 docker-compose MVP 具备准生产试运行能力。

| 方向 | 任务 |
| --- | --- |
| 配置与密钥 | 拆分 dev/staging/prod 配置；引入强随机 `OSB_PLUS_CREDENTIAL_SECRET`；禁止生产 password grant；明确所有必填环境变量 |
| 认证授权 | 完成 Casdoor/OIDC 生产配置手册；校验 issuer、audience、JWKS 缓存刷新；补齐管理员 API 权限测试 |
| 数据库 | 增加迁移回滚策略；补充索引评审；明确备份、恢复、保留周期和演练步骤 |
| API 契约 | 为管理面和 OpenSandbox 兼容 API 增加 OpenAPI 导出、示例请求和错误码规范 |
| 质量门禁 | 建立 CI：后端测试、lint、前端构建、docker build、compose config、基础 API smoke test |
| 运行手册 | 补齐启动、停止、升级、回滚、常见故障、Casdoor 初始化和数据清理手册 |

阶段验收：

- CI 在 main 分支稳定通过。
- staging 环境可从空数据库一键启动并通过 `verify-local.ps1` 等价验收。
- 生产配置不依赖 demo 账号和 password grant。
- 管理员禁用 key、过期 key、无权限访问、跨用户沙箱访问均有测试覆盖。

## 阶段 2：安全加固

目标：满足企业内部安全评审和生产审计要求。

| 方向 | 任务 |
| --- | --- |
| 凭据安全 | key 仅保存哈希；支持 key 轮换、禁用、过期；增加泄露后吊销流程；敏感字段不进日志 |
| 权限模型 | 明确平台管理员和 Agent 用户的最小权限；管理 API 增加操作级审计；高风险操作支持二次确认或审批预留 |
| 网络边界 | OpenSandbox、数据库、Redis 仅内网访问；控制面入口走 TLS；生产禁用默认端口直接暴露 |
| 供应链 | 固定基础镜像版本；生成 SBOM；依赖漏洞扫描；镜像签名与镜像仓库准入 |
| 输入防护 | 请求体大小限制、超时限制、代理路径校验、端口访问限制、沙箱 metadata 限制 |
| 审计合规 | 审计记录覆盖用户、凭据、沙箱生命周期、管理员变更；支持按用户、key、sandbox 查询 |

阶段验收：

- 安全扫描无高危未处理项。
- 所有敏感配置来自 secret 管理，不写入仓库和镜像。
- 审计记录可还原 key 申请、沙箱创建、管理员禁用的完整链路。

## 阶段 3：可靠性与可观测

目标：平台出现异常时可发现、可定位、可恢复。

| 方向 | 任务 |
| --- | --- |
| 健康检查 | 区分 liveness/readiness；检查数据库、Redis、Casdoor、OpenSandbox backend 连通性 |
| 指标 | 暴露 Prometheus metrics：请求量、错误率、延迟、凭据数量、沙箱数量、创建失败率、OpenSandbox 转发失败率 |
| 日志 | 统一结构化日志；贯穿 request_id、user_id、credential_id、sandbox_id；日志脱敏 |
| Trace | 对管理 API、兼容 API、OpenSandbox 转发增加分布式追踪 |
| 告警 | 建立 SLO：API 可用性、P95 延迟、沙箱创建成功率、数据库连接池、Casdoor/JWKS 异常 |
| 故障恢复 | 沙箱状态定期 reconcile；OpenSandbox 与控制面状态不一致时自动修复或标记 |

阶段验收：

- dashboard 能展示当前平台健康状态和关键容量指标。
- 人为断开 Casdoor/OpenSandbox/PostgreSQL 时有明确告警和错误提示。
- request_id 能从用户请求追踪到内部 OpenSandbox 调用。

## 阶段 4：Kubernetes 生产部署

目标：从 docker-compose 升级到可扩缩、可滚动发布、可回滚的 k8s 部署。

| 方向 | 任务 |
| --- | --- |
| 部署形态 | 提供 Helm Chart 或 Kustomize；支持 staging/prod values；保留 docker-compose 作为本地开发环境 |
| 服务拆分 | 初期仍可单 Deployment；当任务调度和状态同步压力上来后拆出 Worker/Scheduler |
| 发布策略 | rolling update、readiness gate、数据库迁移 job、失败自动回滚策略 |
| 资源治理 | CPU/memory request/limit；HPA；连接池上限；OpenSandbox backend 容量保护 |
| 存储依赖 | PostgreSQL/Redis 使用托管服务或高可用部署；明确备份恢复方案 |
| 入口治理 | Ingress/TLS、WAF 或网关、限流、CORS、可信代理头配置 |

阶段验收：

- 空集群可通过文档部署 staging。
- 升级镜像不会中断已有 API 请求超过约定窗口。
- 回滚到上一版本可执行且有演练记录。

## 阶段 5：企业治理与运营能力

目标：支撑多个业务团队长期使用和平台运营。

| 方向 | 任务 |
| --- | --- |
| 配额治理 | 默认配额、用户级配额、并发数、TTL、CPU/memory、创建频率限制 |
| 成本视图 | 按用户、Agent、key、沙箱状态统计资源消耗和趋势 |
| 生命周期策略 | 空闲回收、最大存活时间、异常沙箱清理、批量禁用 key |
| Console 增强 | 搜索、筛选、分页、批量操作、风险提示、操作确认、审计详情 |
| 组织接入 | 支持企业 OIDC 配置模板；用户角色映射规则可配置 |
| API 稳定性 | 版本化管理 API；向后兼容策略；变更日志和升级指南 |

阶段验收：

- 平台管理员可以通过 Console 完成日常运营动作。
- 配额策略可以阻止单个 Agent 用户耗尽平台容量。
- 审计和成本数据能支持月度运营复盘。

## 推荐近期任务清单

优先级从高到低：

1. 建立 GitHub Actions CI，覆盖测试、lint、前端构建、docker build。
2. 整理生产环境变量矩阵，移除生产中的 demo/password grant 假设。
3. 补齐 OpenAPI 导出和错误码规范。
4. 增加安全测试：跨用户访问、禁用 key、过期 key、错误 audience、错误 issuer。
5. 增加结构化日志和 request_id 全链路透传。
6. 设计 k8s Helm/Kustomize 目录，但暂不强拆服务。
7. 增加管理员危险操作确认和审计详情页面。
