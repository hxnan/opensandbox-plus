# OpenSandbox Plus 企业级生产就绪演进路线

目标：把当前 MVP 演进为企业级生产环境 ready 的云沙箱控制面，具备安全可信、稳定可靠、可观测、可扩展、可运维、可治理的交付能力。

## 演进原则

| 原则 | 说明 |
| --- | --- |
| 先稳后拆 | 当前保持单服务交付，先补齐生产能力；当容量、隔离或发布节奏真正需要时再拆 API/Worker/Scheduler |
| 安全默认开启 | 认证、授权、审计、密钥保护、网络边界和最小权限作为默认设计，不作为后置增强 |
| 兼容优先 | Agent 面向 `/v1/sandboxes` 的 OpenSandbox 兼容 API 保持稳定，避免破坏已有 Agent 集成 |
| 集群可扩展 | 控制面需要能管理多个 OpenSandbox 集群，并根据健康、容量、区域和镜像状态做调度 |
| 自动化验收 | 每个阶段都必须沉淀可重复运行的 CI、部署检查和回归脚本 |
| 运维可解释 | 平台状态、审计、告警和故障处置流程要能支持值班人员快速定位问题 |

## 阶段 1：生产基线

目标：让当前 docker-compose MVP 具备准生产试运行能力。

| 方向 | 任务 |
| --- | --- |
| 配置与密钥 | 拆分 dev/staging/prod 配置；引入强随机 `OSB_PLUS_CREDENTIAL_SECRET`；禁止生产 password grant；明确所有必填环境变量 |
| 认证授权 | 完成 Casdoor/OIDC 生产配置手册；校验 issuer、audience、JWKS 缓存刷新；补齐管理员 API 权限测试 |
| 数据库 | 增加迁移回滚策略；补充索引评审；明确备份、恢复、保留周期和演练步骤 |
| API 契约 | 为管理面和 OpenSandbox 兼容 API 增加 OpenAPI 导出、示例请求和错误码规范 |
| 质量门禁 | 建立 CI：后端测试、lint、前端构建、docker build、compose config、基础 API smoke test |
| 运行手册 | 补齐启动、停止、升级、回滚、常见故障、Casdoor 初始化和数据清理手册 |

阶段验收：

- CI 在 main 分支稳定通过。
- staging 环境可从空数据库一键启动并通过 `verify-local.ps1` 等价验收。
- 生产配置不依赖 demo 账号和 password grant。
- 管理员禁用 key、过期 key、无权限访问、跨用户沙箱访问均有测试覆盖。

## 阶段 2：安全加固

目标：满足企业内部安全评审和生产审计要求。

| 方向 | 任务 |
| --- | --- |
| 凭据安全 | key 仅保存哈希；支持 key 轮换、禁用、过期；增加泄露后吊销流程；敏感字段不进日志 |
| 权限模型 | 明确平台管理员和 Agent 用户的最小权限；管理 API 增加操作级审计；高风险操作支持二次确认或审批预留 |
| 网络边界 | OpenSandbox、数据库、Redis 仅内网访问；控制面入口走 TLS；生产禁用默认端口直接暴露 |
| 供应链 | 固定基础镜像版本；生成 SBOM；依赖漏洞扫描；镜像签名与镜像仓库准入 |
| 输入防护 | 请求体大小限制、超时限制、代理路径校验、端口访问限制、沙箱 metadata 限制 |
| 审计合规 | 审计记录覆盖用户、凭据、沙箱生命周期、管理员变更、集群配置变更、镜像上传和镜像分发 |

阶段验收：

- 安全扫描无高危未处理项。
- 所有敏感配置来自 secret 管理，不写入仓库和镜像。
- 审计记录可还原 key 申请、沙箱创建、管理员禁用、集群变更和镜像分发的完整链路。

## 阶段 3：可靠性与可观测

目标：平台出现异常时可发现、可定位、可恢复。

| 方向 | 任务 |
| --- | --- |
| 健康检查 | 区分 liveness/readiness；检查数据库、Redis、Casdoor、各 OpenSandbox 集群和镜像仓库连通性 |
| 指标 | 暴露 Prometheus metrics：请求量、错误率、延迟、凭据数量、沙箱数量、创建失败率、集群健康、镜像同步状态、OpenSandbox 转发失败率 |
| 日志 | 统一结构化日志；贯穿 request_id、user_id、credential_id、sandbox_id、cluster_id、image_id；日志脱敏 |
| Trace | 对管理 API、兼容 API、OpenSandbox 转发、镜像上传和镜像分发增加分布式追踪 |
| 告警 | 建立 SLO：API 可用性、P95 延迟、沙箱创建成功率、数据库连接池、Casdoor/JWKS 异常、集群不可用、镜像同步失败 |
| 故障恢复 | 沙箱状态定期 reconcile；OpenSandbox 与控制面状态不一致时自动修复或标记；镜像分发失败可重试和回滚状态 |

阶段验收：

- dashboard 能展示当前平台健康状态、关键容量指标、各集群状态和镜像分发状态。
- 人为断开 Casdoor/OpenSandbox/PostgreSQL/镜像仓库时有明确告警和错误提示。
- request_id 能从用户请求追踪到内部 OpenSandbox 调用和镜像分发任务。

## 阶段 4：多 OpenSandbox 集群与镜像管理

目标：让 OpenSandbox Plus 从单 backend 控制面升级为多 OpenSandbox 集群控制面，并统一管理沙箱镜像的上传、审核、分发和可用性。

### 多 OpenSandbox 集群管理

| 方向 | 任务 |
| --- | --- |
| 集群注册 | 支持平台管理员新增、编辑、禁用 OpenSandbox 集群；记录 endpoint、内部 server key、区域、标签、容量和健康检查配置 |
| 集群状态 | 周期性采集各集群健康、版本、容量、沙箱状态分布、最近错误和镜像仓库连通性 |
| 调度策略 | 创建沙箱时按集群健康、容量水位、区域、runtime profile、镜像可用状态和配额选择目标集群 |
| 故障切换 | 集群不可用时停止新流量；已创建沙箱保持归属；可配置手动迁移或重建策略预留 |
| 权限与审计 | 集群新增、禁用、server key 更新、调度策略变更都进入审计，并仅平台管理员可操作 |
| Console | 增加集群列表、详情、健康状态、容量、最近错误、手动禁用/恢复和调度权重配置 |

### OpenSandbox 镜像管理

| 方向 | 任务 |
| --- | --- |
| 镜像上传 | 平台管理员可在 Console 手动上传镜像包或登记外部镜像源；支持名称、版本、架构、runtime profile、风险等级和描述 |
| 镜像仓库 | 为每个 OpenSandbox 集群配置对应镜像仓库地址和凭据；凭据按 secret 管理并做最小权限控制 |
| 自动分发 | 镜像上传完成后生成分发任务，自动推送到所有启用的 OpenSandbox 集群镜像仓库 |
| 同步状态 | 记录每个镜像在每个集群的状态：pending、uploading、available、failed、disabled；失败保留错误原因和重试次数 |
| 可用性约束 | Agent 创建沙箱时只能选择已通过策略且在目标集群可用的镜像；调度器避免把请求路由到缺少镜像的集群 |
| 版本治理 | 支持默认镜像、推荐版本、禁用版本、风险标记、灰度分发和回滚到上一可用版本 |
| 审计追踪 | 镜像上传、分发、禁用、删除、重试和策略变更全部记录审计 |

阶段验收：

- 可通过 Console 注册至少两个 OpenSandbox 集群，并看到各自健康和容量。
- 平台管理员上传镜像后，系统自动推送到各集群镜像仓库，并展示每个集群的同步状态。
- 某个集群镜像同步失败时，不影响其他集群完成分发；失败集群可手动重试。
- 创建沙箱时调度器只选择健康、容量充足且目标镜像可用的集群。
- 集群禁用、镜像禁用、镜像分发失败等场景都有审计记录和清晰错误信息。

## 阶段 5：企业治理与运营能力

目标：支撑多个业务团队长期使用和平台运营。

| 方向 | 任务 |
| --- | --- |
| 配额治理 | 默认配额、用户级配额、集群级配额、并发数、TTL、CPU/memory、创建频率限制 |
| 成本视图 | 按用户、Agent、key、集群、镜像、沙箱状态统计资源消耗和趋势 |
| 生命周期策略 | 空闲回收、最大存活时间、异常沙箱清理、批量禁用 key、按集群清理异常资源 |
| Console 增强 | 搜索、筛选、分页、批量操作、风险提示、操作确认、审计详情、集群和镜像运营视图 |
| 项目大屏首页 | 增加对外展示型首页，用于介绍 OpenSandbox Plus 的定位、价值、架构能力、兼容 OpenSandbox API、Agent key 自助申请、多集群和镜像管理能力 |
| 宣传视觉资产 | 结合文生图能力生成项目背景图、宣传图、场景图和功能示意图，用于首页 hero、能力板块和发布材料 |
| 组织接入 | 支持企业 OIDC 配置模板；用户角色映射规则可配置 |
| API 稳定性 | 版本化管理 API；向后兼容策略；变更日志和升级指南 |

阶段验收：

- 平台管理员可以通过 Console 完成日常运营动作。
- 配额策略可以阻止单个 Agent 用户、单个镜像或单个集群耗尽平台容量。
- 审计和成本数据能支持月度运营复盘。
- 访问项目首页时，无需登录即可理解项目价值、核心能力、部署形态和下一步接入路径。
- 首页背景图和宣传图有统一视觉风格，并可随版本发布迭代。

## 推荐近期任务清单

优先级从高到低：

1. 建立 GitHub Actions CI，覆盖测试、lint、前端构建、docker build。
2. 整理生产环境变量矩阵，移除生产中的 demo/password grant 假设。
3. 补齐 OpenAPI 导出和错误码规范。
4. 增加安全测试：跨用户访问、禁用 key、过期 key、错误 audience、错误 issuer。
5. 增加结构化日志和 request_id 全链路透传。
6. 设计 OpenSandbox 集群管理数据模型、API 和 Console 页面。
7. 设计镜像上传、镜像仓库配置、自动分发任务和同步状态模型。
8. 设计项目大屏首页的信息架构、视觉方向和文生图资产生成流程。
9. 增加管理员危险操作确认和审计详情页面。
